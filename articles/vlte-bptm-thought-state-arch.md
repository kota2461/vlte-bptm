> この記事は個人開発中のアーキテクチャ **VLTE-BPTM v1.6（alpha）** の設計思想と、現時点で測れている性能を「限界込み」で共有するものです。完成品の発表ではありません。むしろ「ここまで測れて、ここはまだ未確立」という現在地そのものを記事の主題にしています。

## TL;DR

- 自然言語入力を **64bit の Thought Code（ルーティングキー）** に変換し、「どう考えるか」を決める軽量な前段レイヤを作っている。
- 設計の核は **責務の分離**: 意味理解 / 処理経路の選択 / Unit実行 / 回答生成 を、別々のコンポーネントとして契約で区切る。Router を知識モデルや回答モデルに肥大化させない。
- 効果がはっきり出ているのは **「LLMを呼ぶ前に呼ばないと決める」** こと。挨拶のような入力では重いLLM呼び出し（4〜6秒）を丸ごと回避でき、桁違いに速い。
- 一方で **回答本文の正しさ（semantic answer quality）の独立評価は現状0件** で、未確立。route選択精度も、分布内では高いが交差評価では落ちる。この記事ではそこを隠さない。

## 1. 動機: 「全部LLMに投げる」への違和感

いまのLLMアプリの主流は、生のプロンプトをそのままモデルに渡し、分類も判断も生成も一つのモデルにやらせる構成です。これは強力ですが、設計者として2つ気になる点がありました。

ひとつは **コスト配分の粗さ**。「こんにちは」も「この設計を多角的に検証して代替案を3つ出して」も、同じ重いモデルに同じように流れる。前者に推論予算を割く必要はないはずです。

もうひとつは **判断の不透明さ**。「なぜこの入力をこう処理したのか」がモデルの内部に溶けていて、ログとして外から追えない。

VLTE-BPTM は、この2点に「LLMの前段に薄い思考状態レイヤを置く」というアプローチで答えようとしています。

## 2. 全体像

処理は一本のパイプラインとして流れます。

```text
input
  -> active_bits          # 入力から立つビット
  -> selected_units       # どの思考Unitを使うか
  -> inhibition integration
  -> horizontal mesh      # Unit同士の投票・優先度
  -> optional vertical stack
  -> optional hybrid stack mesh
  -> action_vector        # 行動の重みベクトル
  -> llm_order            # LLMへの指示書
  -> optional external runtime executor
```

設計上もっとも大事にしているのは、次の責務境界です（Semantic Routing Architecture v0.1 より）。

1. 意味抽出モデルは **答えを生成しない**。
2. 処理Routerは **生プロンプトを解釈しない**（検証済みの意味Packetだけを受ける）。
3. Coreは **意味内容を生成しない**。Unitと実行順序を構成するだけ。
4. 回答LLMが知識推論と文章生成を担当する。
5. 不確実性は隠さず、再解析・clarify・上位の処理クラスへ送る。
6. 出力や評価結果を **自動学習に使わない**。

ポイントは、それぞれが「やらないこと」で定義されていることです。Router に知識を持たせない、Core に文章を書かせない。肥大化を契約で禁じることで、各層を独立に評価・差し替えできるようにしています。

## 3. 実際に動かすとどうなるか

`この設計をレビューしてください` を入れたときの実際の出力（抜粋）がこれです。

```jsonc
{
  "thought_code": { "value": "0x3E70C407151C4398", "width": 64, "role": "external_routing_key" },
  "metrics": { "active_bit_count": 26, "active_bit_density": 0.40625, "selected_unit_count": 3 },
  "action_vector": {
    "reply": 0.713, "ask": 0.0, "explain": 0.254, "plan": 0.256,
    "verify": 0.368, "summarize": 0.0, "creative": 0.0, "caution": 0.295
  },
  "llm_order": {
    "mode": "verify",
    "instruction": "Check assumptions, risks, and evidence before concluding.",
    "metadata": { "horizontal_mesh_winning_axis": "verify", "threshold_profile": "deep_verify_v1" },
    "constraints": [
      "Do not persist cloud LLM output as training data.",
      "Do not perform automatic learning or automatic model updates."
    ]
  }
}
```

「レビューして」という入力に対し、Horizontal Mesh の勝者が `verify` になり、最終的にLLMへ渡す指示書（`llm_order`）が「結論の前に前提・リスク・根拠を確認せよ」という verify モードの命令になっています。**どの入力が、どのビットを立て、どのUnitを勝たせ、どんな指示書になったか**が全部JSONで追える。この追跡可能性が動機の2つ目（不透明さ）への回答です。

なお `0x3E70C407151C4398` のような Thought Code はあくまで**外部ルーティングキー**で、意味座標ではありません。Unit内部の `16×16` も意味マップではなく活性バッファです（ここを誤解されやすいので仕様で明記しています）。

## 4. 性能: 出ているところと、出ていないところ

ここが本記事の肝です。区別すべき指標が3つあります。

### 4.1 速度: 「呼ばない」が一番効く

前段で「これはLLM不要」と判断できる入力は、重いモデル呼び出しを丸ごとスキップします。実測（ローカルの gemma-4-12b に対する router あり/なし比較）:

| 入力 | LLM直接 | Router経由（direct） | 倍率 |
| --- | ---: | ---: | ---: |
| こんにちは | 4.86 s | 0.0053 s | ≈ 900x |
| ありがとう！ | 6.71 s | 0.0002 s | ≈ 27,000x |

この数字は **「Router自体が速い」のではなく「LLMを呼ばずに済ませた」** ことによるものだと正直に書いておきます。モデルが賢くなったわけではありません。Adapter自体のオーバーヘッドは1呼び出しあたり 0.0002〜0.186 ms 程度で、判断レイヤとしては十分軽量です。

### 4.2 route選択精度: 分布内は高い、交差評価は落ちる

凍結した回帰スイートでは高い値が出ます。

- frozen regression: 25/25 = **1.00**（raw accuracy, macro-F1 1.0）
- foundation anchors: 58/58 = **1.00**

ただしこれは「学習・調整に使った分布の内側」での話です。別系統のfixtureへ交差適用すると落ちます。

- Core を Router境界fixtureへ適用: 9/25 = **0.36**（macro-F1 0.364, 95% Wilson 0.20–0.56）
- 非groupedのholdout: 26/30 = **0.867**（Wilson 0.70–0.95）
- 反復CV: **0.769**（点が非独立のため信頼区間は出していない）

固定fixtureの100%を「一般的な精度」として宣伝してはいけない、というのを自分への戒めとして仕様書に明記しています。

### 4.3 回答品質: 未確立（ここが正直な現在地）

そして最も重要な但し書きです。

```text
independent_case_count = 0
status = not_established
```

**回答本文そのものの正しさを独立にblind評価したデータは、現時点で0件です。** 測れているのは主に経路選択の精度であって、「出てきた答えが正しいか」はまだ確立していません。ここを誤魔化すと記事全体の信頼が崩れるので、最初のTL;DRから明示しています。

## 5. 何に使えるか（と、まだ使えないこと）

**今すでに価値がある使い方:**

- LLM前段の **コストゲート**。「呼ぶ/呼ばない」「軽いモデル/重いモデル」を決定的なルールで振り分ける。
- 入力処理の **監査ログ**。どの入力がどう判断されたかをJSONで完全に追える。
- 指示書（`llm_order`）の **明示的な制約注入**（例: クラウド出力を学習に使わない、を構造的に強制）。

**まだ任せられないこと:**

- 回答の正しさを保証すること（独立評価が0件）。
- 学習分布の外側でのroute精度（交差評価で0.36まで落ちる帯がある）。

つまり現状の正確な立ち位置は「**回答を賢くする装置」ではなく「LLMをいつ・どう呼ぶかを決める、追跡可能な前段レイヤ**」です。

## 6. これから

設計は「一つのRouterに全部集約しない」方向で3トラックに分けて独立評価していきます。

1. **Pattern Language Model**: 生文から最小意味信号を抽出（答えは作らない）
2. **Processing Router**: 意味Packetから処理経路と予算を決める（生プロンプトは見ない）
3. **LLM Integration**: 既存Core / Runtime / tool を接続して全体品質と費用を評価

直近の最優先は、4.3で空欄になっている **回答品質の独立blind評価を立ち上げること** です。そこが埋まって初めて「性能」と胸を張れる、と考えています。
