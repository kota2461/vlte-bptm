## 1. プロジェクト概要(30秒版)

VLTE-BPTM は、自然言語入力を観測可能な処理段階に分解し、外部LLM向けの命令
(`llm_order`)を組み立てる**ルーティング・命令生成パイプライン**である。
思考内容そのものは生成しない。最重要の理念は次の3つ。

1. **観測可能性**: 全段階をJSONで追跡できる(schema/pipeline version + 6段階trace)
2. **責務分離**: Thought Code 64bit(外部ルーティングキー)と Pattern Unit `C=64`
   (内部feature channel)は数値が同じだけで別物
3. **自動学習禁止**: 人間が承認したPattern DBエントリのみが学習対象。
   学習は常に明示的な`train`操作。クラウドLLM出力の教師データ化は禁止

正本仕様は `docs/VLTE_BPTM_v1_0a.md`、版別計画は `docs/VLTE_BPTM_roadmap.md`。

### 3.2 カリキュラム学習(Pattern Lab経由・境界遵守)

フローは常に「pending投入 → ユーザー指示に基づく明示的承認 → 明示的train」。
承認操作の記録は `build/approve_math_curriculum.py` /
`build/approve_language_curriculum.py` に残してある。

- **数学** (`seed-math`, 86件): number_sense → addition(答えが8になる
  同回答バリエーション: 3+5/5+3/6+2/4+4/7+1+言い換え+文章題) → elementary →
  junior_high → high_school(二次関数・三角関数・指数対数・数列・帰納法・
  ベクトル・微積分・確率統計)
- **言語基礎** (`seed-language`, 42件): greetings_ja / greetings_en(模範応答文を
  `thought_form.candidates`に保持) / basic_ja(主語述語・助詞・敬語・因果) /
  basic_en(翻訳・時制・文法検証)

### 3.3 学習結果(最終状態)

- Pattern DB: **128 patterns** / routes = build:11, clarify:12, explore:9,
  respond:75, summarize:9, verify:12
- モデル: `build/pattern_router_model.json`
  (`pattern-router.model.v1`, averaged_perceptron, epochs=40)
- 検証精度: **91.3%** (23件ホールドアウト)。学習履歴はDBの`training_runs`に5回分
- 挨拶テスト: 未学習バリエーション10/10が`respond`、模範応答の取り出しも確認
  (`build/greeting_response_test.py`)
- 数学テスト: 未学習入力12問中11問正答

### 3.4 既知の弱点(未解決)

1. **`explore` Routeが最少(9件)**。「8になる別のたし算を考えてください」級の
   境界事例は`respond`に落ちることがある。→ 承認済みexplore例の追加と再学習で改善
2. **確信度が未較正** (`confidence_calibrated: false`)。ルーターを信じるか
   clarifyへフォールバックするかの閾値が未定義
3. ルーターは文字n-gram+語トークンの線形分類であり**意味理解ではない**。
   実トラフィックでの評価は未実施(Phase 2/3が前提)
4. respond:75 と多数派偏重。少数Routeを増やす際はカリキュラム追記→`seed-*`→
   承認→trainの順(差分だけ挿入される。UNIQUE制約で重複は自動スキップ)

## 5. 定型操作チートシート

```powershell
cd "D:\Thought State Register"
python -m pytest -q                          # 49件パスが正常状態
```

`python build\approve_math_curriculum.py` の形式(過去の記録を参照)。

## 6. 絶対に守ること(理念の再掲)

- 自動学習・自動重み更新・自己改変を実装しない
- クラウドLLM出力を教師データとして保存しない(Phase 2でテスト固定予定)
- 未評価・却下候補を学習集合に入れない
- v1.0a Coreの受け入れfixture(`tests/fixtures/v1_0a_cases.json`)を壊す変更は
  契約変更としてドキュメント更新とセットで行う
- 推測でUnit Type・係数・スキーマを固定しない(評価fixtureと設計ログを根拠にする)
