## 1. 追加発見 — 否定表現テンプレートとラベルの交絡(最重要)

レビュー依頼1への回答の核心。48件の本文を調べた結果:

| Route | 否定表現あり(「〜は作らず」「〜は挙げず」「Do not」「without」等) | なし |
| --- | ---: | ---: |
| respond | **12 (12/12 = 100%)** | 0 |
| explore | 5 | 7 |
| build | 0 | **12** |
| verify | 0 | 6 |
| clarify | 1 | 5 |

**respondラベルは否定マーカーの有無と完全相関している。** このまま48件を
承認・学習すると、モデルは「意図の差」ではなく「ず、/Do not → respond」という
表層ショートカットを学習する可能性が高い。対照ペア設計自体は優れている
(話題・言語・難度を固定して意図だけを変える)が、「意図を変える手段」が
ほぼ常に否定メタ指示で実現されているため、ペア内の識別信号が単一の表層特徴に
集中してしまった。

なお現48件の**評価値(27/48)自体は汚染されていない**(モデルは否定表現を
まだ学習していない)。問題は承認・学習後のラウンドで生じる。

### 修正提案(承認前に実施)

1. respond側12件のうち6〜8件を、否定メタ指示なしの自然な肯定形へ書き換える
   (例:「実装手順は作らず、このAPIが何をするものか説明してください」→
   「このAPIは何をするものですか」)。
2. 逆方向の否定を少数追加する(例: build側に「説明だけで終わらせず、
   作業手順を組んでください」)。目標は否定マーカーのRoute間分布を概ね均等化。
3. Round 2では「メタ指示なしの自然文」を主体にする(実ユーザーは
   「チェックリストは作らず〜」とは言わない)。

## 2. レビュー依頼10項目への回答

### Q1. Round 1設計にラベルリーク・過度な誘導はないか

- 学習データとの完全一致0件は確認済み(リークなし)。
- ただし上記§1の否定交絡が「過度に明示的な誘導」に該当する。respond全件が
  否定メタ指示で構成され、表層手がかりがラベルを完全分離する。承認前の
  書き換えを推奨。
- 副次論点: 各境界内でテンプレートが反復している(br系respondは全て
  「〜は作らず、…を説明してください」)。`contrast_group`単位のgrouped splitだけ
  ではテンプレートリークを防げない。`template_id`の付与を推奨(Q5参照)。

### Q2. build/respond、clarify/verifyのRoute定義自体に曖昧さがないか

ある。判定規則の明文化を推奨する。

- **build vs respond**: 曖昧地帯は「手順の説明」。「手順を説明して」(理解が
  目的=respond)と「手順を組んで」(実行可能な成果物が目的=build)は日本語で
  近接する。**「成果物基準」での定義を推奨**: ユーザーの状況に対して実行可能な
  順序付きタスク/計画/チェックリストを*作る*ならbuild、対象の理解を返すなら
  respond。トピック語(API、実装、移行)はRoute判定に使わない。
- **clarify vs verify**: 日本語「確認してください」が両義(確認の質問=clarify /
  検証=verify)。**「材料の有無基準」を推奨**: 判定対象(URL・数値・前提)が
  入力に提示されていればverify、欠けていて先に取得が必要ならclarify。
  cv系ペアはこの基準と整合しており、定義文書化だけが残作業。
- この2基準(成果物基準・材料基準)をRoute仕様書に決定木として固定し、
  レビュアー間の一致度(Cohen's κ)を少数サンプルで一度測ると良い。

### Q3. AI支援synthetic promptを非蒸留とする整理に不足はないか

整理は概ね妥当。teacher出力・logit・hidden reasoningを使わず、ラベルが
Route仕様+人間承認に由来する以上、一般的なteacher-student蒸留とは異なる。
不足は2点:

1. **作成モデルの記録がない。** `authoring_model`(モデル名・バージョン)、
   `authoring_date`、生成セッション識別子をsource metadataへ追加すべき。
   「AI支援」とだけ書かれた候補は、後から由来を監査できない。
2. 「非蒸留」整理は技術分類であり、**生成物を学習データに使うこと自体への
   各プロバイダ利用規約の適用は別問題**(レポート自身の注記通り)。承認時に
   `license_name`を`project-original-synthetic`のまま使うなら、規約確認の
   実施日と判断をmetadataに残すと監査が閉じる。

### Q4. 承認前に追加すべきprovenance / license / review metadata

現状の`ai_assisted_authoring`等に加えて:

- `authoring_model` / `authoring_date`(Q3)
- `template_id`: ペアを跨いで共有される文型の識別子(Q1・Q5の前提)
- `reviewer`(人間識別子)と`review_round`: 既存`reviews`テーブルはverdictと
  notesのみで誰が見たか残らない
- `label_provenance`: 参照したRoute仕様の節(例: `route-spec.v2 §build`)
- `eval_eligible: false`フラグ: 「学習に入れた48件は評価へ再利用しない」
  (レポート§8判断3)をデータ側で機械的に強制する

### Q5. grouped splitのgroup key設計

`contrast_group + template + source`で概ね正しい。具体化すると:

- **必須**: `contrast_group`(ペアを分割しない。ペアの片割れがtrainに居ると
  validationの残り半分が近似重複になる)
- **必須**: `template_id`(§1の通り、文型はgroupを跨いで反復している。
  現状これがメタデータに存在しないため付与が先)
- `source`は現48件では全件同一なので実効なし。複数カリキュラム混在時に効く
- 推奨key: `(source_url, contrast_group)`を最小単位とし、`template_id`の
  近似重複検出(文字n-gram Jaccard、v0.2予定)で**分割を跨ぐtemplate共有を
  検出したらレポートに警告**を出す。全templateを同一foldへ寄せると
  fold構成が崩壊するので、検出→書き換え(データ側の修正)を優先する。

### Q6. sealed setの件数と配分

目的を「pre/post比較で56%→75%級の改善を検出」とすると:

- 対応のある比較(同一セットでpre/post、McNemar検定)なら、
  **境界ごとに20対照ペア(40文)以上、4境界で計160文**が実用最小。
  これで境界別±20ポイント級の変化が概ね判別できる。48件級では
  Wilson区間が±13ポイントあり、境界別(12件)では±25ポイント超で
  実質判定不能(今回のレポート自身が示した通り)。
- v0.2記載の「各Route最低30件」と整合させると**150〜200文**が妥当。
  配分: 日英50/50、route偏りなし(respond過半にしない)、
  simple/compound半々、否定マーカーはRoute間で均等(§1の教訓)。
- 作成規律: 学習カリキュラムと**別作成者・別セッション・別テンプレート**、
  凍結日とSHAを記録、開封は1回(v0.2契約通り)。
  反復調整用には別途grouped validationを同規模で持つ。

### Q7. 英語25%: データ追加とモデル変更のどちらが先か

**データが先。** 根拠:

- 承認済み英語は約20件でうち非respondは7件程度。モデルは英語の
  build/clarify/verify/summarizeをほぼ見たことがなく、英語16件の予測が
  respond14/explore2に潰れたのは**クラス事前分布の支配**であって、
  特徴量の構造的失敗の証拠はまだない。
- 推奨: 英語の非respondを各Route 15〜20件(計60〜80件)承認して再学習し、
  英語sealed sliceで再測定。それでも英語が日本語より20ポイント以上
  低いままなら、初めてモデル側(言語別n-gram範囲、単語bigram追加等)を検討。
- 安価な並行施策: `text_features`のchar n-gramは空白除去後の2-4gramなので
  英語では単語境界情報が落ちる。単語unigramは既にあるため、
  **単語bigram追加**は線形モデルのまま試せる低リスク変更。

### Q8. 文字n-gram線形モデルを維持する合理性と変更基準

維持する合理性は現時点で強い:

- データが166件しかなく、より大きいモデルは過学習する
- 決定的(固定seed)・JSON可読・依存ゼロ・監査可能 — プロジェクトの
  観測可能性と自動学習禁止の理念に直接適合する
- 56%の主因はデータ(言語・Route偏り、境界例不足、§1の通り意図表現の
  多様性不足)で説明可能であり、モデル限界と断定する材料がまだない

変更を検討すべき判断基準(すべて満たしたとき):

1. Route×言語で各50件以上の承認データを揃えた後も、grouped validationの
   macro-F1が目標(例: 0.75)に届かない
2. 誤り分析で「語彙が同じで意図だけ違う」失敗が支配的(否定・照応・暗黙意図
   など、n-gramで原理的に表現できない差)
3. 言語特徴の追加(否定フラグ、Route別意図動詞辞書、疑問マーカー等の
   明示特徴)でも改善しない — 線形のまま特徴を足す方が先

### Q9. fallback閾値を境界別・Route別へ変えるべきか

**今は変えない。** 理由:

- 現較正は166件のrepeated k-fold由来で、Route別に割ると1ビンあたりの
  実効サンプルが一桁になり、閾値自体が雑音になる
- 今回の失敗(正しいbuild 3件をclarifyへ退避)は閾値の粒度ではなく
  **較正分布と評価分布のずれ**が原因。先にv0.2のgrouped calibration setで
  再較正するのが正順
- Route別閾値はv0.3の条件(「十分な評価件数がある場合だけ」)が正しい。
  その前に**コスト行列の明文化**(不要な聞き返しコスト vs 誤回答コスト)を
  推奨。閾値の最適点はコスト比で決まるもので、現在は暗黙にコスト比1:1で
  raw accuracyを見ている

### Q10. CoreとPattern Router統合判断に必要な追加証拠

不足している証拠は4つ:

1. **共通sealed混合セットでの直接比較**: 現状は「各自のfixtureで強く、
   相手のfixtureで弱い」という交差結果のみで、母集団を代表する共通評価が
   ない。sealed set(Q6)にCore代表系の入力クラスも混ぜ、同一セットで
   Core / Pattern Router / 統合候補を測る
2. **統合契約の定義**: 「統合」が(a) Router高確信時にCore modeを上書き、
   (b) RouterをCore Selectorの一特徴量として注入、(c) 入力クラスで分岐、
   のどれかで必要な証拠が変わる。先に契約を1つ選び、shadow比較する
3. **shadow mode観測**: v1.2のprivacy storeを使い、現行Core経路のまま
   Router予測を**ログのみ**並走させ、実入力分布での一致率・不一致時の
   どちらが正しいかを人間レビューでサンプリング評価する。これが
   最も安価で強い証拠になる
4. **回帰安全性**: 統合ONでもCore受け入れfixture(8件+v1.1系)が全通過する
   こと、および統合OFFへの即時ロールバック手段

## 4. 結論

- レポートは数値・手法とも誠実で、再現性は完全に確認できた。
  56.25%を現行実力値とする自己評価も妥当(むしろ既存fixture 25/25を
  一般化精度と区別している点は模範的)。
- 唯一の重大な見落としは**否定表現とrespondラベルの完全相関**(§1)。
  これを修正せずに承認・学習すると、Round 1の投資がショートカット学習で
  毀損され、次のsealed評価で「改善が再現しない」結果になる可能性が高い。
- 構造変更(モデル置換・閾値細分化・Router統合)はいずれも時期尚早で、
  データ修正 → grouped/sealed評価基盤 → 再較正 → shadow観測の順が正しい。
