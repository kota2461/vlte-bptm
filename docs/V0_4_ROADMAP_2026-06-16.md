# v0.4 Roadmap — from external review (2026-06-16)

Captures an external-agent review of the frozen v0.3 first model and the
agreed direction. v0.3 stays FROZEN
(`docs/INTENT_MODEL_v0_3_FROZEN_2026-06-16.md`); everything here is SUCCESSOR
work and must obey the standing discipline (no auto-learning, human-approved
labels only, campaign never a training gate, no LLM output as training data,
critical_underprocessing stays 0, no regression on v0.2 strengths).

## Review verdict (agreed)

v0.3 is a healthy first freeze: it broke the v0.2 markers ceiling (intent
0.80→0.86, e2e 0.76→0.82, critical 0) and — more valuable than the number —
established the **evaluate/freeze/inherit discipline**. Not trusting the
learned layer (markers-first; learned only on no-match + margin≥0.15) and
stopping the synthetic grind were the right calls.

## Phased plan

### v0.3.1 — production hardening (do FIRST; mostly no model change)
1. **real dev slice** (100–150 real, human-approved, disjoint from BOTH
   campaign and training; weak-category-heavy; ja-dominant + a little mixed) —
   for threshold selection, NOT a training set. *(pending — needs real data)*
2. **fix gate margin on the dev slice, not the campaign** (removes the
   campaign-informed concern about the 0.15 gate). *Successor-only — changing
   the live margin changes frozen v0.3 behaviour. (pending)*
3. **observability logging**: persist `decided_by` / margin / top scores on
   every route. **DONE 2026-06-16** — `route()` returns `RouteResult.trace`
   (decided_by markers/learned/fallback + intent_margin + top_scores +
   adapter_version 0.3.1); `route_and_execute`/ExecutionResult expose
   `decided_by`/`margin` (incl. `direct`); server `/route` returns trace.
   Decisions are byte-identical (no-regression test + identical campaign
   by-category numbers). Frozen model artifact untouched.
4. **weak-category failure-log auto-aggregation**. **DONE 2026-06-16** —
   `build/observability_report.py` (by-category e2e/intent + by-decided_by +
   per-miss decided_by/margin/top-scores). First run: fallback 5/9 (4 misses
   sit at margin 0.007–0.098, just under the 0.15 gate), learned 3/3, markers
   33/38 — the exact coordinates for targeted real-data collection.
5. **rollback-able model registry** (already largely present via
   `intent_deployment` history; make it explicit). *(pending)*

### v0.4 — structural intent resolution
1. **Temporal Normalizer v0.1** — a THIRD deterministic preprocessor (not
   markers, not learned). Emits `has_temporal_reference` / `temporal_kind`
   (now_validity / recent_past / near_future / deadline / vague_future) /
   `routing_hint` as AUXILIARY evidence for verify/explore — must NOT override
   the safety-first ordering.
2. **compound_primary_resolver v0.1** — small deterministic resolver above the
   learned layer: from intent_candidates + verb/deliverable/conditional spans,
   pick primary + secondaries + prerequisites. Axes: deliverable→build/
   summarize; "問題ないか/正しいか"→verify; "なぜ/仕組み"→explain;
   "考えたい/案を広げたい"→explore; "どれ/何が不足"→clarify.
3. **marker strong/weak/safety split** — merge rule: safety→deterministic;
   strong-single→marker; none→learned if margin ok; weak→learned/resolver if
   margin ok; multiple-non-safety→compound resolver.
4. **explicit-vocab perceptron** for observability (reversible vocab + top
   features per prediction) BEFORE any embedding move.
5. real-log approved data pipeline strengthening (already built; extend tags).

### v0.5 — model upgrade (only after v0.4)
Prefer A/B (explicit-vocab linear / ngram logistic) to keep observability;
C (small sentence embedding + shallow classifier) later; avoid D (distilled
tiny encoder) — it muddies the observable thesis.

## Successor data target: weak-category-targeted, multi-tag (not just intent)
Label schema beyond intent: `category` (temporal/compound/indirect…),
`surface_style` (direct/indirect/casual/mixed/short), `primary_rule`
(deliverable/prerequisite/question_focus/missing_info). Used by eval / data
collection / review UI even if the model ignores most of it.

## New test sets to add (anchors)
- Temporal idioms: 「これは今でも使えますか？」「最近の仕様でも大丈夫？」「今度これを直したい」「あとで見返せるようにまとめて」「前に話した案はまだ有効？」
- Compound primary: 「確認して、問題なければ改善案を出して」「改善案を出す前に問題点を見て」「これを要約して、最後に次の手順も作って」「まず妥当性を見て、必要なら修正して」
- Indirect explanation: 「ここが腑に落ちません」「なぜそうなるのかピンと来ない」「何が違うのか分からない」「この挙動の理由が見えない」
- Casual JP short: 「これいけます？」「なんか変？」「どう直す？」「使えるやつ？」「つまり？」

## Reviewer caveats / my notes (Claude)
- **Strongly agree**: hardening-first, real dev slice as the single most
  important fix, temporal normalizer + compound resolver as deterministic
  modules, explicit-vocab before embeddings, multi-tag corpus, the design-doc
  acceptance NOTE (added 2026-06-16).
- **Caveat — the real dev slice is itself the same data bottleneck.** It needs
  real, human-approved, weak-category examples; it cannot be synthesized
  (falsified + boundary). It is smaller and measurement-only, so sourceable
  incrementally — but it is real work, not free.
- **Caveat — deterministic resolvers can become a brittle rule pile** (the same
  marker ceiling). Validate the temporal normalizer and compound resolver on a
  held-out set, never fit to the campaign or dev slice.
- **Caveat — "0.82→0.86–0.88 realistic" is a hypothesis, not a promise.** Treat
  every structural fix as measure-once-on-dev-slice, don't fit.
- **Naming**: v0.4 = "markers + learned fallback + intent-structure resolution"
  (temporal + compound + strong/weak/safety). Reasonable.
